# 메신저 양방향 결재 — 제품 차별화 방향

> **2026-06-21 제품 방향 인터뷰 산출물.** 가설·방향 문서이며 확정 구현 스펙이 아니다.
> 차별화 전략과 1순위 MVP(Discord 양방향 결재)를 정의하고, 후속 레버(AI·인사이트)를 잇는다.
> 구현 착수 시 본 문서를 PRD로 발전시키고 SYSTEM_DESIGN/FEATURE_LIST/ERD와 동기화한다.

## 1. 포지셔닝

> **"flex만큼 근태·HR 되면서, 한국형 전자결재를 메신저에서 끝낸다."**

AbleWork의 구조적 자산은 `HR 이벤트(휴가·근태·일정) → 전자결재 문서 자동 생성($transaction) → 메신저 알림`이 **한 줄로 흐른다**는 점이다. 경쟁군은 이 조합을 갖지 못한다.

| 경쟁군 | 강점 | 빈틈 |
|---|---|---|
| 카카오워크·네이버웍스 결재 | 결재 UX, 메신저 네이티브 | 그 뒤의 HR·근태·잔액·정책 데이터 없음 |
| 더존·이카운트 | 데이터 완비 | 무겁고 비싸고 메신저 비친화 (50~300인엔 과잉) |
| **flex·시프티 (← 1차 전장)** | 근태·연차·전자계약 | **전자결재의 깊이가 얕음** |

## 2. 전략 종합

| 축 | 결론 |
|---|---|
| **1차 전장** | flex·시프티(HR SaaS) — 약점 = 전자결재의 깊이 |
| **타겟 고객** | 사무직 중심 중소기업(IT·서비스·본사), 고정 근무, 메신저 친화 |
| **차별화 시퀀스** | ① 메신저 양방향 결재 → ② AI 업무비서 → ③ 데이터 인사이트 |
| **기본 체력** | 도입·운영 경량화(임포트·템플릿)는 차별화가 아니라 flex와 *동급으로 맞춰야 할* 전제 |

AbleWork는 이미 한국형 결재 문화(전결·대결·전단계반려·부서협조/수신·공람·문서대장·다결재자 병렬)를 Phase 2에서 깊게 구현했다. flex가 못 가진 이 깊이가 무기이며, 이를 **메신저에서 끝내는 경험**으로 증폭한다.

## 3. 1순위 MVP — Discord 양방향 결재 `[승인][반려]`

> 플랫폼 우선순위: **Discord 먼저**(이미 webhook 알림 자산 보유 → 봇 확장 최단, PoC·IT/스타트업 고객). 진짜 타겟인 **카카오워크·네이버웍스는 2차** — 따라서 처음부터 `MessengerProvider` 인터페이스로 추상화한다.

### 3.1 사용자 시나리오
1. 직원이 휴가/기안 상신 → (기존) HR 이벤트 → 문서 자동 생성 → 알림 발행
2. 결재자 Discord에 **"결재 요청" 메시지 + `[승인] [반려]` 버튼**
3. `[승인]` → 즉시 처리 → 메시지가 `✅ 승인 완료`로 갱신 + 다음 결재자에게 알림
4. `[반려]` → 코멘트 입력(모달) → 반려 처리 → 메시지 갱신

### 3.2 기존 자산 재사용
- `NotificationListener`가 `NOTIFIABLE_EVENTS`(SSOT) 구독 → Discord/email/in-app 디스패치. 수신자 = `assigneeId ?? drafterId ?? requesterId`.
- 결재 액션 API 완비: `POST /requests/:id/approve`(요청연동 문서 APPROVER_R1), `POST /documents/:id/steps/:stepId/approve|reject`.
- 감사로그(audit) 인프라 존재 → 메신저 결재 행위 기록에 재사용.

### 3.3 신규 컴포넌트 (필요한 것만)
1. **Discord Bot**(webhook → Bot 토큰) + **Interaction 수신 엔드포인트**(서명 검증)
2. 알림 메시지를 **버튼 포함 Bot 메시지**로 전환 — `custom_id`에 `documentId/stepId` 인코딩
3. **메신저 계정 ↔ employee 매핑** 테이블(최초 1회 연동) + 버튼 클릭자 == step assignee 검증
4. 버튼 콜백 → **기존 결재 액션 API 호출** → 메시지 갱신
5. **`MessengerProvider` 추상화** — Discord는 첫 구현체, 카카오워크·네이버웍스로 교체 가능하게

### 3.4 MVP 범위
- ✅ 포함: 휴가·일반 기안의 `[승인][반려]`(반려 코멘트), 메시지 상태 갱신, 본인 검증 + 감사로그
- ⏸ 제외(후속): 결재문 열람·진행현황, 기안 작성·상신, 전결·대결·협조 등 복합 액션(→ 웹 딥링크 유도), 카카오워크·네이버웍스

### 3.5 리스크 · 설계 주의
- **법적 효력(본인성)**: "누가 눌렀나" — 계정 매핑 + 감사로그로 방어, 민감 문서는 딥링크 2차 확인 옵션.
- **타겟 불일치**: Discord는 사무직 주류가 아님 → **PoC/IT 고객용**. 검증 후 카카오워크·네이버웍스가 진짜 타겟이므로 `MessengerProvider` 추상화 필수.
- **NEVER 목록 정합**: 생체인증/2FA 등 Enterprise 인증은 범위 밖 — 본인성은 계정 매핑 + 감사로그 + (옵션) 딥링크로 해결.

### 3.6 로드맵 (3주 PoC)
- ✅ **W1** (PR #68): Bot 인프라(Interaction 수신·서명검증) + `MessengerProvider` 추상화 — 프로덕션 PING/PONG 실연결 확인.
- ✅ **W2**: 계정 연동 API(PR #71) + 콜백→결재 액션·메시지 갱신(W1) + **상신 시 현재 결재자에게 `[승인][반려]` DM 발송**(`MessengerApprovalListener`) → 양방향 결재 회로 완성.
- ⬜ **W3**: 본인 검증 강화·에러·E2E·실데모 문서 → **PoC 데모**
- 직후: 2순위 AI를 `[AI 요약]` 버튼으로 결합 / 카카오워크 Provider 확장

## 4. 후속 레버 — ② AI 업무비서 / ③ 데이터 인사이트

### ② AI 업무비서 — 결재 요약·초안 (메신저 봇 통합)
인터뷰 결론: AI의 1순위 형태는 **결재 요약·초안**, 포지션은 **메신저 봇에 녹임**.

→ **1순위 메신저 결재와 한 몸**이 된다. 결재 요청 메시지에 `[AI 요약]` 버튼 → 결재자가 긴 문서/첨부를 **3줄 요약**으로 보고 `[승인]`. 기안자에겐 양식+맥락 기반 **초안 생성**.

- **차별화 연료**: "결재 + HR 결합 데이터"(flex·카카오워크엔 없음)를 컨텍스트로 주입 — 신청자 잔액·근태 이력·정책 등을 요약/초안에 반영.
- **모델**: 최신 Claude로 요약·초안 품질 확보. 결재문/첨부 텍스트 + HR 맥락을 프롬프트에.
- **핵심 컨셉**: §3의 메신저 봇이 곧 **"AI 결재 허브"** — 알림 + AI 요약 + 승인/반려가 한 메시지에서 끝난다. 이것이 "메신저에서 결재 끝낸다"의 완성형.
- **시퀀스**: 메신저 결재 MVP(§3.6 W1~W3) 직후, `[AI 요약]` 버튼을 봇 메시지에 결합하는 단계로 진행.

### ③ 데이터 인사이트
AI 축의 연장. 근태+결재 결합 대시보드, 조직 건강도, 야근/휴가 패턴, 이탈 위험 신호. **자연어 리포트 질의**("지난달 개발팀 야근 1위?")는 ②의 봇 자연어 인터페이스 확장으로 후속 검토.

## 5. 미해결 · 결정 필요
- 메신저 계정 연동 방식: OAuth vs 관리자 일괄 매핑
- 민감 문서 판단 기준(딥링크 2차 확인 트리거)
- `MessengerProvider` 인터페이스 경계(메시지 전송/버튼/콜백/계정매핑 추상화 수준)
- AI 요약 컨텍스트 범위(어떤 HR 데이터까지 프롬프트에 주입할지 — 개인정보 경계)
- AI 비용/지연 정책(요약 호출량·캐싱·동기/비동기)
